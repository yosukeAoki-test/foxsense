# FoxSense 親機-子機通信 LoRa化 設計書

最終更新: 2026-07-01

> **決定事項（2026-07-01、モジュールは後日更新）**
> - **無線ルート = B（技適優先）**。日本での実運用を前提に、技適済みモジュールで構成する。**試作/R&D段階**。
> - **無線モジュール = E220-900T22S(JP)（CLEALINK/EBYTE, 技適UART, SX1262系, ~¥1,540）を親機・子機とも採用**。当初ES920LR3案だったが**高価(~¥4,378)のためE220に変更**（技適UARTのまま約1/3。秋月/スイッチサイエンスで入手容易）。
> - **子機 = ESP32-C3(XIAO) + E220(UART)**（親機とツールチェーン統一）。
> - **親機 = LILYGO T-SIM7080G-S3 を維持し、E220 を UART 増設**（LTEは温存, M0/M1でモード制御）。
> - **実装済み**: 共通ドライバ `src/e220.h`（透過モード, 0xA5フレームを自前フレーミング, RSSIバイト=raw-256）。親機 `src/`、子機 `foxsense-lora-child/`。センサは親子とも **FS304-SHT3x(防水温湿度I2C)**。
> - ⚠️ 素SX1262(Ra-01SH ~¥300)は最安だが技適無し→量産で工認を取るなら将来ルートA。今は技適E220で進める。§3.4参照。

## 1. 目的とスコープ

親機-子機間の無線を **TWELITE(2.4GHz ISM) → LoRa(920MHz帯)** に置き換える。

狙い:
- **通信距離・障害物透過性の向上**（2.4GHz → サブGHz。圃場・ハウス越し・地形での到達性が大幅改善）
- 見通し外・広域配置での子機取りこぼし低減

上位の**アプリ層プロトコル（`0xA5…0x5A` フレーム / XORチェックサム / `parentIdHash` / ペアリング / logical ID）は無線非依存なので原則そのまま流用**する。置き換えるのは物理層（無線モジュールとその駆動コード、タイミング設計）のみ。

### スコープ外（今回やらない）
- LoRaWAN（ゲートウェイ/NWKサーバ運用）への移行 — 親子直結の現構成には不要
- 子機のOTA
- メッシュ/マルチホップ（LoRaは素P2P、スター型のみ）

## 2. 現状（TWELITEの何を置き換えるか）

| レイヤ | 現状(TWELITE) | 対応ファイル |
|--------|--------------|-------------|
| 親機 無線接続 | ESP32-S3 UART(GPIO43/44,115200) + wake(GPIO2→DIO0) | `src/config.h:40-47`, `src/main.cpp:499-518` |
| 親機 wake送信 | `sendMWXWakeTrigger()` / v2 wakeパケット | `src/main.cpp:525-574` |
| 親機 データ収集 | 30秒窓で `0xA5…0x5A` をパース | `src/main.cpp:579-745` |
| 親機 ペアリング | pair送信/ACK待ち | `src/main.cpp:1120-1195` |
| 子機A | **TWELITE DIP(JN5169)＝MCU+無線一体** + AHT21B / MWX | `foxsense-twelite-child/FoxSenseChild.cpp` |
| 子機B | ATmega328P + BME280 + UART(TWELITE前提) | `child-firmware/src/` |
| プロトコル定義 | `0xA5`ヘッダ等の定数 | `src/config.h:59-66`, `child-firmware/src/config.h:13-20` |

**最重要ポイント:** 子機Aの TWELITE DIP は **MCU と無線が一体**。LoRa化するとこの一体型が消えるため、子機は必ず **「ホストMCU + LoRaモジュール」の2チップ構成**に作り替えになる。子機Bの ATmega328P 資産が活きる。

## 3. ハードウェア選定（★決定が全体を左右する）

### 3.1 大前提：日本での実運用には「技適(工事設計認証)」が必須

920MHz帯は電波法の規制対象。**技適マーク付きモジュールを使えば、機器を組み込んでもそのまま合法運用できる**。技適の無い素モジュール（自分で認証を取らない限り）は、電源を入れて電波を出した時点で違法になり、圃場に設置できない。よって**技適の有無がモジュール選定の第一基準**。

さらに日本の920MHzは **ARIB STD-T108**（キャリアセンス/LBT・送信時間・デューティ比の制限）に従う必要がある。**技適済みUARTモジュールはこのLBT等を内部で自動処理**してくれるので、ファーム側で法令タイミングを作り込まなくて済む（これが大きな採用理由）。

### 3.2 候補比較

| モジュール | チップ | I/F | 技適 | 消費電力 | 特徴 | 概算単価 |
|-----------|--------|-----|:---:|--------|------|--------|
| **ES920LR3**（EASEL）★推奨 | STM32WL(RF+MCU一体) | **UART** | **○** | 超低（スリープµA級） | 日本設計・ARIB内部処理・P2Pモード・現UART構成にほぼドロップイン | ~¥2-3k |
| ES920LR2（EASEL） | SX1276+MKL16Z | UART | ○ | 低 | LR3の一世代前・ピン互換 | ~¥2-3k |
| E220-900T22S（クレアリンク版） | SX1262 | UART | ○ | 低 | 安価・国内技適版が流通 | ~¥1-1.5k |
| TLM922S-P01A（Kiwi） | — | UART | ○ | 低 | 国内技適LoRa | ~¥1-2k |
| IM920sL（インタープラン） | 独自 | UART | ○ | 低 | 技適・長距離だが**LoRaではなく独自方式**（`0xA5`は載るが将来のLoRa資産流用不可） | ~¥2k |
| Ra-01SH（Ai-Thinker） | SX1262 | SPI | **×** | 超低・CAD等フル制御可 | 安価・OSSライブラリ豊富だが**技適無し→そのまま国内不可** | ~¥300-500 |
| 素SX1262/RFM95 | SX126x/127x | SPI | **×** | フル制御可 | 同上、要自己認証 | ~¥300-800 |

### 3.3 推奨：**ES920LR3（技適UARTモジュール）を親機・子機ともに採用**

理由:
1. **技適済み** → そのまま圃場に設置できる（実運用の必須条件をクリア）。
2. **UART接続** → 親機は現行のTWELITE UART配線・シリアル処理構造をほぼ流用（＝改修最小）。子機もATmega328Pの既存UARTコードを転用できる。
3. **ARIB/LBTを内部処理** → ファームで法令タイミングを実装しなくてよい。
4. **STM32WL一体で超低消費** → 子機の電池寿命に有利。

トレードオフ（後述§5で吸収）:
- ES920はUARTデータパイプ型で、CADや超長プリアンブル等の**低レベル無線制御は使えない** → ランデブー（親子の起床同期）方式は module が出来る範囲＝「タイマ間欠 or 子機起点プッシュ」で設計する。

### 3.4 参考：素SX1262（ルートA / 今回は不採用）

CAD/長プリアンブルによる**最省電力なwake-on-radio**、**BOMコスト最優先**、または **ESP32-C3/S3+SX1262一体ボード(Heltec/T3/XIAO+Wio)で素RadioLibを叩く**構成が技術的には最良。この場合、親機も **LILYGO T-SIM7080G-S3 に素SX1262(Ra-01SH等)をSPI増設**して同一スタックに揃える（カメラ未使用なら全IO空き、GPIO35-37は不可）。

**今回は不採用**：これらは技適が無く、日本でそのまま設置すると違法。採用するには (a) 自社で工事設計認証(工認)を取得、(b) 実験試験局免許、(c) R&D/試作限定、のいずれかが前提。**将来 工認まで踏み込むなら**、§5のランデブーを「CAD wake」に置換して電池寿命を最大化できる。

> 以降の本設計は **決定=ルートB / ES920LR3(技適UART)** を前提に記述する。

## 4. 全体アーキテクチャ

```
        [子機] ESP32-C3 + AHT21B/BME280 + ES920LR3(UART)
                   │  LoRa 920MHz P2P
                   ▼
        [親機] ESP32-S3 + ES920LR3(UART, 旧TWELITE配線を流用)
                   │  LTE (SIM7080G, 既存)
                   ▼
              サーバ (既存, 変更なし)
```

- LoRaは **P2P（素の1対多スター）**。LoRaWANは使わない。
- 親機ES920 ⇔ 子機ES920 は同一 RFパラメータ（周波数ch, SF, BW, CR, プリアンブル長, ネットワーク識別子）に**事前設定**しておき、P2Pモードで**ペイロードをそのまま透過**。ここに既存 `0xA5…0x5A` フレームを載せる。
- `parentIdHash` を引き続きペイロードに含めるので、**近隣に別の親子ペアが居ても混信を論理的に排除**できる（現行v2/v3の仕組みをそのまま活用）。

### 4.1 データフロー方式：2案

現行TWELITEは「親が15秒wakeビーコン連投 → 子は間欠受信で拾って応答」。LoRaでは15秒連続ビーコンはデューティ/送信時間の観点で不利。以下2案から選ぶ。

**方式A：親トリガ＋子間欠受信（現行の移植・推奨）**
- 親: サイクル開始時に wakeビーコンを **LBT内蔵送信で数回（例: 1秒間隔×N）** 送出 → その後 RX を30秒開く。
- 子: `間欠受信`（例 9s スリープ / 1s 受信）で wakeビーコンを待つ。拾ったら測定→**logical ID をスロットにした時分割**で data 応答（衝突回避）。
- 長所: 時刻同期不要で堅牢。**既存のペアリング/wake/collect フローとコードをほぼ流用**。
- 短所: 子が定期的にRX ON（現行TWELITEと同等の電力プロファイル）。

**方式B：子起点プッシュ（省電力最優先・将来）**
- 子: 自タイマで定期起床→測定→data送信→（任意でACK受信）→スリープ。RX ほぼ不要で最省電。
- 親: 自サイクルの受信窓（例60秒）で待ち受け。ACK内に時刻を載せて子のドリフト補正。
- 短所: 親子の**時刻ランデブー設計**が必要（ATmega328PのWDTはRC発振で誤差大 → ACK毎の再同期 or 外部32kHz水晶が要る）。

→ **まず方式Aで移植**（改修最小・確実）。運用が固まったら方式Bで電池寿命を伸ばす、の二段構え。以降は方式A前提。

## 5. タイミング・電力・法令設計（LoRaで一番変わる所）

### 5.1 RFパラメータ（初期推奨）
- 周波数: 920MHz帯の1ch（ES920のch番号で指定。親子で一致）
- 帯域幅 BW: 125kHz
- 拡散率 SF: **SF7〜SF9**（近〜中距離。まずSF7で運用、届かなければ上げる）
- コーディングレート CR: 4/5
- ペイロード: 最大21バイト（現v3 dataフレーム）→ LoRaの上限255に対し余裕

参考エアタイム（BW125, CR4/5, 21B前後）:
- SF7 ≈ 約40-50ms / SF9 ≈ 約120-160ms / SF12 ≈ 約1s超

→ **SFを上げるほど届くがエアタイムが伸び、デューティ制約と電力が増える**。初期はSF7で始め、現地で必要最小限だけ上げる。

### 5.2 法令（ARIB STD-T108）
- LBT（キャリアセンス）・送信時間上限・デューティ制限は **ES920が内部処理**するので、ファーム側は「送りたいデータをUARTに書く」だけでよい。
- ただし**wakeビーコン連投や高SFの多用はモジュール内部の送信制限に当たり、送信が間引かれ得る**。方式Aのビーコン回数・SFは、モジュールの送信可否を見ながら控えめに設定する。

### 5.3 タイミング定数の見直し（`config.h`）
現行TWELITE値はLoRaのエアタイムに合わせて再設定が必要。

| 定数 | 現行 | LoRa初期案 | 備考 |
|------|------|-----------|------|
| wakeビーコン窓 | 15s連続400ms間隔 | 数回×1s間隔 | LBT/デューティ配慮 |
| 子 SLEEP/LISTEN | 9s/1s | 9s/1.2s程度 | LISTENは1ビーコン+プリアンブルを確実に含む長さに |
| 子応答スロット | なし(即応) | logicalID×スロット幅(例200ms) | LoRa衝突回避のTDMA |
| CHILD_RESPONSE_TIMEOUT | 30000 | 子数×スロット幅+マージン | エアタイム分を加味 |
| PAIRING_RESPONSE_TIMEOUT | 10000 | 10000前後 | エアタイム分を加味 |

## 6. ファーム側設計

### 6.1 親機（`src/`）
方針: **TWELITE用のUART送受信・パース・ペアリングのロジックは温存**し、下記だけ差し替える。

- `config.h`
  - `TWELITE_*` ピン定義を `LORA_*` にリネーム／流用（**物理ピンGPIO43/44/2はそのまま転用可能** → 配線変更最小）。
  - LoRa RFパラメータ定数（ch/SF/BW/CR/プリアンブル/ネットワークID）と§5.3のタイミング定数を追加。
  - `0xA5` 系プロトコル定数は**変更なし**。
- `initTwelite()` → `initLora()`
  - UART begin は流用。加えて**ES920の初期化AT設定**（P2Pモード・RFパラメータ・データ透過モード）を投入し、設定はモジュール内保存（毎回投入 or 一度書き込み）。
- `sendMWXWakeTrigger()/sendWakeSignal()` → **wakeビーコンを ES920 のデータ送信**として送る（`0xA5…0x5A` の wakeフレームをUARTへ）。DIO0直叩きのMWXトリガは廃止。
- `collectChildData()` / パース群 → **ロジックそのまま**。ES920受信データ（P2P透過）から `0xA5…0x5A` を切り出す。受信フレームに付随する**RSSIはES920の受信メタから取得**（現行のRSSIバイトに詰め替え）。
- ペアリング送受信 → **ロジックそのまま**（フレームをES920経由で送受）。

> ES920はP2Pモードで受信データを一定書式（バイナリ/ASCII）でUART出力する。**バイナリ透過**に設定し、既存パーサへそのまま流す。受信メタ(RSSI等)の書式はモジュール仕様に合わせて1箇所で吸収する。

### 6.2 子機（新規ディレクトリ推奨：`foxsense-lora-child/`）
- ベースは **ESP32-C3 + AHT21B/BME280 + ES920LR3(UART)**。PlatformIO(espressif32)で親機とツールチェーン統一。
  - ※SX1262一体ボード(Heltec/T3/XIAO+Wio)は技適無しのため不採用。C3は素のC3(XIAO ESP32-C3 / C3 SuperMini等)＋ES920を使う。
  - ※センサは親機v3が気圧対応。AHT21B(温湿のみ)採用なら pres は0/未送で送る（現行の版分岐で吸収）。
  - **代替**: 最省電力/低コスト最優先なら既存 `child-firmware/` の **ATmega328P + ES920**（既存コード資産を流用。EEPROM保存やstate machineをそのまま活かせる）。
- 状態機械（FACTORY_DEFAULT→PAIRED→測定→送信→スリープ）と **不揮発保存（magic/parentIdHash/logicalID/checksum）は流用**（C3ではNVS/Preferences、ATmega版は`child-firmware/src/config.h:26-30`のEEPROMレイアウト）。
- 無線処理:
  - `listenForWakeSignal()`: ES920をRXにし、間欠受信でwakeフレーム待ち。
  - wake受信→`parentIdHash` 照合→測定→**logicalIDスロット待ち→dataフレーム送信**。
  - `handlePairingRequest()`: pairフレーム照合→保存→pair ACK返信。
- 省電力: ES920スリープ（LR3は超低µA）＋ESP32-C3 deep sleep（~5µA級、UART/GPIOでwake）。ATmega版はWDTスリープ。
- **子デバイスID**: TWELITEのHWシリアルが使えなくなるため、**ESP32-C3のefuse MAC下位4バイトからchildIdを生成**（採番が自動・一意で楽）。ATmega版はシグネチャ生成 or 製造時EEPROM書込。サーバのデバイス登録もこのIDに合わせる。

> 子機A(MWX/TWELITE DIP)版は**廃止**（無線一体チップのため移行不可）。子機はESP32-C3+ES920（or ATmega328P+ES920）へ一本化。

### 6.3 影響を受けないもの
- LTE/SIM7080G通信、サーバ送信、OTA、IR制御 — **無関係・変更なし**。
- サーバAPI（config/pairing-result）— §7参照、原則そのまま。

## 7. サーバ / API 影響

原則 **変更なし**（プロトコルが無線非依存のため）。ただし運用面で以下を確認/軽微調整:

- `parentIdHash` / `children[].childId` / `logicalId` の仕組みは**そのまま利用**。
- **子デバイスIDの採番方法が変わる**（TWELITE HWシリアル → ATmega側生成/書込ID）ため、子機の**新規登録・在庫管理の運用**を合わせる。
- 気圧を送らない子機構成にする場合、サーバ側の pres 欠損許容を確認。
- ペアリング結果 `POST /api/devices/config/:deviceId/pairing-result` は流用。

## 8. 移行手順（運用）

1. **評価機で疎通**: 親ES920×1・子ES920×1を卓上で P2Pリンク確立（RFパラメータ合わせ→`0xA5`フレーム往復）。
2. 親機ファーム: `initLora()` と wake/collect のES920差し替え → 既存パーサでdata取得を確認。
3. 子機ファーム: ATmega328P+ES920でペアリング→測定→送信の一巡を確認。
4. 距離試験: 現地でSF7から始め、到達しない区間だけSFを上げて最小構成を決定。
5. **技適表示・アンテナ**: 技適済みモジュールでも、指定アンテナ・利得条件を守る（モジュール仕様の範囲内で使用）。
6. 全機展開。TWELITE系（親DIP・子DIP・MWXファーム）は撤去。

## 10. 既知の注意点 / 論点

- **技適が採用可否を決める**。素SX1262(Ra-01SH等)は安く高機能だが国内そのまま不可。技適UART(ES920等)が実運用の既定路線。
- **子機は必ず2チップ化**（TWELITE DIPの一体無線が失われるため）。子機A(MWX)は移行不可＝廃止。
- LoRaはTWELITEより**低速・エアタイム大**。15秒ビーコン連投はそのまま持ち込まず、方式A（間引きビーコン＋TDMA応答）へ。
- **時刻同期が要る方式B（子起点プッシュ）は電池寿命に最良だが、MCUスリープタイマのRC誤差対策（C3のdeep sleep timerも±数%、ATmega WDTも同様 → ACK再同期 or 外部32kHz）が前提**。まずは方式Aで確実に。
- ES920はP2Pデータパイプ型のため**CAD/長プリアンブルwakeは不可**。最省電のwake-on-radioが要件なら素SX1262＋自己技適の検討に戻る。
- プロトコル層(`0xA5`/`parentIdHash`/ペアリング/logicalID)は**無線非依存で全面流用**。ここは作り直さない。
- 子デバイスIDの採番が変わる（HWシリアル → 生成/書込）。**登録運用の変更を忘れない**。
